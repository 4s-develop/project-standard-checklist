# 基本設計 チェックリスト（詳細版）

## 1) 基本設計書（NFR 実装方針）

- 目的: システム横断の基本方針（アーキ、NFR 対策）を定義
- 手順: アーキ概要/依存/NFR 対策（キャッシュ/リトライ/タイムアウト/レート制限）を記載
- ツール: 基本設計書.docx
- 成果物: 基本設計書
- 受入基準: 主要構成/NFR 対策明確、未決課題は記載
- 証跡: 文書 + レビュー記録
- 役割: TL、承認=PM

## 2) 画面一覧/画面遷移図

- 目的: UI 範囲と動線の可視化
- 手順: 画面 ID・目的・遷移条件を一覧化し、遷移図を描く
- ツール: Figma/draw.io
- 成果物: 画面一覧.xlsx / 遷移図
- 受入基準: 主要動線が図示、例外遷移あり
- 証跡: 図/一覧 + レビュー
- 役割: FE、承認=TL

## 3) API 一覧・IF 仕様（OpenAPI ドラフト）

- 目的: API 仕様を一元化
- 手順: エンドポイント/リクエスト/レスポンス/エラーコード（日本語メッセージ方針）を記載
- ツール: openapi.yaml
- 成果物: API 仕様ドラフト
- 受入基準: 主要 API 定義、スキーマ整合、エラー方針あり
- 証跡: openapi.yaml + レビュー
- 役割: BE、承認=TL

## 4) システム構成図（C4）

- 目的: コンテキスト/コンテナ/デプロイ構成の共有
- 手順: C4 の Context/Container/Deployment を作図（WAF/SSL/監視も反映）
- ツール: draw.io/PlantUML
- 成果物: C4 図
- 受入基準: 依存/ネットワーク/セキュリティ経路が明確
- 証跡: 図 + レビュー
- 役割: TL/SRE

## 5) セキュリティ設計書

- 目的: 認証/認可/RBAC/秘密管理/CORS/入力/出力バリデーションを明確化
- 手順: 方式/ポリシー/実装責務を記載
- ツール: Security-Design.docx
- 成果物: セキュリティ設計書
- 受入基準: 攻撃面（XSS/CSRF/SQLi）への対策記載、WAF 前提
- 証跡: 文書 + レビュー
- 役割: TL/BE/FE、承認=PM

## 6) ADR（主要決定）

- 目的: 技術選定の根拠を記録
- 手順: 背景/決定/代替案/影響をテンプレに記載
- ツール: docs/adr/ADR-00X.md
- 成果物: ADR
- 受入基準: 重要決定ごとに ADR あり、PR で 2 名承認
- 証跡: ADR + PR リンク
- 役割: TL/PM

## 7) 品質レビュー記録

- 目的: 指摘/改善点の管理と承認
- 手順: テンプレで観点/指摘/期限/担当を記録 → 承認
- ツール: quality-review-record.docx
- 成果物: レビュー記録
- 受入基準: 重大指摘クローズ、承認済
- 証跡: 記録 + 議事録
- 役割: TL/PM/QA
